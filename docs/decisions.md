# 技术决策记录（ADR）

> 只追加，不改写。推翻旧决策时新增一条并注明取代关系。

## D-001 通用模板而非每书定制（2026-07-12）

**决策**：阅读器代码与书籍内容彻底分离；每本书是一个符合 schema 的静态数据包，书籍差异（注释体系/目录层级/实体）全部下沉到数据与配置。
**理由**：用户明确希望"导入文本即可获得阅读体验"；对比读通鉴的三个板块可知，纪事本末式重组、主题聚合等高级功能本质是同一段落库的不同索引，段落 id 全局唯一 + 结构化 schema 是一切后续功能的地基。

## D-002 纯前端 + 静态数据（2026-07-12）

**决策**：M1 不做后端；数据为静态 JSON 按章节懒加载；设置/收藏/进度存 localStorage。
**理由**：核心阅读体验不需要服务端；静态托管零成本部署；用户系统推迟到确有云同步需求时（远期）。

## D-003 原文只存繁体，简体实时转换（2026-07-12）

**决策**：数据包中文言原文（含注释原文）存繁体，前端用 opencc-js 按需转简体；白话翻译直接存简体。
**理由**：古籍底本为繁体，繁→简是近乎无损的单向转换，存两份必然漂移；opencc 转换开销在段落级可忽略。

## D-004 实体/注释用字符区间锚定 + 词典自动匹配（2026-07-12）

**决策**：实体 `span=[start,end)`、注释 `anchor` 锚定在原文字符位置；实体 span 不由人工书写，而是维护"实体词典"（sources/entities），由导入脚本在原文中自动匹配（长词优先、不重叠）。
**理由**：人工数偏移量必错；词典跨章节复用，且是未来全书批量标注管线的雏形。渲染端（src/lib/segment.ts）对脏数据容错（越界/重叠跳过），校验端（scripts/validate.mjs）对脏数据零容忍。

## D-005 sources/ 为唯一事实来源（2026-07-12）

**决策**：`public/data/` 的章节文件是生成物，禁止手改；原文、翻译、词典的修改都发生在 `sources/`，跑 ingest 脚本再生成。
**理由**：保证数据可再生、可校验、可批量重构（例如未来 schema 升级只需改脚本重跑）。

## D-006 技术栈 Vite + React + TS，路由用 HashRouter（2026-07-12）

**决策**：如题；测试 Vitest + Testing Library。
**理由**：纯前端 SPA 的最小可行栈；HashRouter 使静态托管（含 file:// 与无 rewrite 的托管）开箱即用，后续要 SEO 再迁 BrowserRouter/SSG。

## D-007 译文自产、不接 API，由 Claude 会话内产出（2026-07-13）

**决策**：白话译文/句级对齐/人物生平全部在 Claude Code 会话内由模型直接产出，不调用外部 LLM API，也不抓取网络译文入库。
**理由**：网络白话译文源自有版权出版物（演绎作品版权），公开上线有风险，且裴注无现成译文、句级对齐仍需加工；已评估 DeepSeek V4 Pro 批量方案费用可忽略（蜀书约 ¥2–10），但用户选择订阅额度内手翻——质量最可控、无密钥管理。将来扩魏书+吴书 50 卷可接 DeepSeek（OpenAI 兼容端点），管线源格式（alignments/persons）已与之兼容。

## D-008 句级对齐源格式用句子字符串对而非区间（2026-07-13）

**决策**：`sources/alignments/` 存 `[{o: "原文句", t: "白话句"}]` 字符串对，由 ingest 脚本按顺序子串定位换算为字符区间，并强校验拼接覆盖完整（失败即报错终止）。
**理由**：人工/LLM 手数字符偏移必错；字符串精确匹配让任何错位在构建期立刻暴露。实测 52 段约 190 句对一次性全部通过。
